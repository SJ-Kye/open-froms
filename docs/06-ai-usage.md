# 06. AI 도구 활용 내역

> 이 문서는 개발 진행에 따라 **실시간으로 갱신됩니다**. "AI 를 썼다" 가 아니라 *어떤 도구로 / 무엇을 / AI 생성물의 무엇을 왜 고쳤는지* 를 구체적으로 기록합니다.

## 사용 도구
| 도구 | 용도 |
|---|---|
| **Claude Code (Opus)** | 설계 문서 작성, 스캐폴딩, 코드 생성/리팩토링, 디버깅, 테스트 작성 |

## 개발 원칙 (AI 협업 방식)
1. **설계 문서 우선**: 코드보다 설계를 먼저 확정하고, AI 에게 그 설계에 맞춰 구현하도록 지시합니다. 확정 전 상세 설계는 `dev-docs/`(로컬 참조)에 두고, 구현·확정된 부분만 `docs/`(커밋 대상)로 옮깁니다.
2. **하이브리드 TDD**: 도메인/서비스 규칙은 실패 테스트를 먼저 작성(red)한 뒤 구현(green)하고 리팩토링합니다. 커밋 히스토리에 red-green 이 남습니다.
3. **AI 생성물 무비판 수용 금지**: 생성된 코드를 검토하여 버그/설계 위반/불필요 코드를 직접 수정하고, 수정 이유를 아래 로그에 남깁니다.

### [Phase 0.6] DB 설계 정련 & V1 스키마 확정

이 단계는 사용자가 요구사항을 질문-응답으로 확정하고 Claude 가 구현·**실측 검증**하는 방식으로 진행했습니다. 설계 판단은 사용자가 내렸고, Claude 는 그 판단을 스키마로 옮긴 뒤 실제 DB 로 검증하여 문서의 주장(예: "폼 삭제 시 연쇄 삭제")이 실제로 성립하는지 확인했습니다.

**T8. 감사 컬럼 도입 (선택적 적용) — Claude Code**
- created/updated_by·at 4종을 기본으로 하되 **모든 테이블에 붙이지 않았습니다.** 적용 기준을 "그 컬럼이 새로운 정보를 담는가" 로 정하고, users·forms 는 4종, responses 는 불변이라 created_* 2종, 상위에 종속되는 questions·answers 등은 제외했습니다. **이유**: 종속 테이블의 감사값은 상위와 항상 같아 중복이며, 최다 행 테이블(answers)에 4컬럼을 더하면 저장 비용만 늘어납니다.
- 생성자/변경자를 FK(users.id) 가 아닌 VARCHAR('ANONYMOUS' 가능)로 두었습니다. **이유**: 익명 응답을 NULL 없이 표현하고, 계정 삭제가 감사 기록을 제약하지 않도록 하기 위함입니다.

**T9. 질문 타입 9종 확장 — Claude Code**
- 기존 5종에 DROPDOWN·NUMBER·DATE·TIME 을 더했습니다. **직접 판단**: 사용자가 "체크박스" 추가를 요청했으나, 체크박스는 기존 MULTIPLE_CHOICE(다중선택)와 저장 구조가 동일하므로 **신규 타입으로 만들지 않고** 그 점을 문서에 명시했습니다. **이유**: 저장 구조가 같은 타입이 둘이면 집계 로직이 중복됩니다.
- RATING/NUMBER 의 척도를 정의할 곳이 없던 문제를 questions.min_value·max_value 로 해결했습니다.

**T10. 응답 저장 모델 단순화 (answer_options 제거) — Claude Code**
- 사용자 제안("answer_options 빼도 될 것 같은데")을 받아 조인 테이블을 제거하고 answers.option_id 로 통합했습니다. **이유**: 순수 조인 테이블이라 가장 흔한 택1형이 값 하나를 저장하려고 행 두 개를 쓰고, 대시보드 핵심 질의인 선택지별 집계에 조인이 끼었습니다. answers 를 "값 1건"으로 재정의하니 N:M 이 1:N 두 개로 분해되어 3NF 를 유지하면서 조인이 사라졌습니다.

**T11. 폼 삭제 연쇄 삭제 버그 발견·수정 — Claude Code (실측 검증)**
- **버그 수정**: answers 의 FK 중 response_id 에만 ON DELETE CASCADE 가 있어, 실제 PostgreSQL 에서 폼 삭제 시 `fk_answers_question` 위반으로 **삭제가 실패**했습니다. question_id·option_id 에도 CASCADE 를 부여해 고쳤습니다.
- **주목할 점**: 이 버그는 **AI 가 생성한 원본 스키마(커밋 ac0668b)에 이미 존재**했고, 문서는 "폼 삭제 시 연쇄 삭제"를 보장한다고 적고 있었습니다. 일회용 PostgreSQL 16 컨테이너에 실데이터를 넣고 `DELETE FROM forms` 를 실행하는 회귀 시나리오로 재현하여 발견했습니다. **교훈**: 문서의 주장을 실측으로 교차검증하지 않으면 AI 생성물의 결함이 그대로 남습니다.

**T12. users.id UUID 전환 (혼합 전략) — Claude Code**
- 사용자 요청으로 users.id 를 UUID(UUIDv7, 앱 생성)로, forms.user_id 를 동일 타입으로 변경했습니다. 나머지 테이블은 BIGINT 를 유지하는 혼합 전략입니다.
- **직접 판단**: 앞서 "전부 UUID" 안을 검토했을 때는 인덱스 지역성 비용을 근거로 전량 기각했으나, **users 한 곳** 은 근거가 다름을 구분했습니다. 순차 PK 가 노출하는 가장 민감한 정보가 가입자 수이고 users 는 저볼륨이라, "이득이 가장 크고 비용이 가장 작은 한 곳" 으로 판단해 여기에만 적용했습니다. **이유**: UUID 를 인가 대체재로 오용하지 않되(IDOR 은 소유권 검사로 막음), 인가로 막을 수 없는 정보 누출은 표적 지점에서 차단하기 위함입니다.
- **검증**: H2·PostgreSQL 16 양쪽에서 UUID PK/FK 동작과, forms.user_id 에 정수 삽입이 타입 불일치로 거부되는 것을 확인했습니다.

**T13. 설계 문서(04) 본문 작성 — Claude Code**
- 사용자 지시로 docs/04 본문을 채우고 V1__init.sql 과 컬럼·인덱스 일치를 스크립트로 대조했습니다. **주의**: 통상 docs/ 설계 본문은 사용자가 직접 작성하는 것이 본 프로젝트 규율이며, 이 초안은 사용자 검토·재작성 대상입니다.

### [Phase 1] 백엔드 공통 인프라 (에러 처리·traceId·API 이력·Security/JWT) — Claude Code

패키지 구조를 **기능별+계층 하위 / 모듈러 모놀리스**로 재편(사용자 선택)한 뒤 공통 인프라를 구축했습니다. 규율은 `common ← user ← form ← response` 단방향 의존이며, 이 경계가 아래 설계 판단들의 근거가 됩니다.

**T14. 응답/추적 형식 결정 — 사용자 판단 + Claude 구현**
- 사용자 요청(공통 응답·traceId·API 호출 이력)을 받아, "에러만 래퍼 + traceId 는 헤더" 로 확정했습니다. **이유**: 성공 응답은 설계 문서(05)대로 bare 본문을 유지해 문서 변경을 최소화하고, `X-Trace-Id` 헤더 + 에러 본문 `traceId` 필드로 추적성을 확보했습니다.
- API 호출 이력(`api_call_logs`, V2)은 **메타데이터만 비동기 저장**하고 본문·비밀번호·JWT 는 저장하지 않습니다. 주체 컬럼을 FK(UUID) 대신 `principal VARCHAR` 로 둔 것은 **직접 판단**입니다. **이유**: 매 요청 email→UUID 조회를 없애고, `common` 이 `user` 모듈에 의존하지 않도록 경계를 지키기 위함입니다(감사 `created_by` 와 동일 소스).

**T15. Boot 4 = Jackson 3 발견·수정 — Claude Code (버그 수정)**
- 필터단 401/403 직렬화에 `com.fasterxml.jackson`(Jackson 2) `ObjectMapper` 를 주입했다가 `NoSuchBeanDefinitionException` 으로 컨텍스트 로드가 실패했습니다. **원인**: Spring Boot 4 는 **Jackson 3(`tools.jackson`)** 을 기본 빈으로 씁니다. import 를 `tools.jackson.databind.*` 로 교체해 해결했습니다.

**T16. 슬라이스 테스트 함정 회피 — Claude Code (설계 판단)**
- `@WebMvcTest` 슬라이스가 `WebConfig→인터셉터→라이터` 빈을 끌어와 로딩에 실패했습니다. `GlobalExceptionHandler` 검증은 애플리케이션 컨텍스트 없이 `standaloneSetup` 으로 전환해 advice 만 독립 검증했습니다. **이유**: 예외 변환 규칙은 컨텍스트 무관하므로 슬라이스 의존을 없애는 편이 견고합니다.

**T17. 모듈 경계로 인한 인증 범위 분리 — Claude Code (설계 판단)**
- Security/JWT 를 `common` 에 두되 **무상태 검증만**(토큰 subject=이메일) 담고, `UserDetailsService`·비밀번호 대조는 Phase 2(`user` 모듈)로 미뤘습니다. **이유**: `common` 이 `user` 에 의존하면 모듈러 모놀리스 경계가 깨지므로, DB 조회 없는 서명 검증만 공통에 남겼습니다.

### [Phase 2] 인증 (회원가입·로그인·본인 조회) — Claude Code (하이브리드 TDD)

서비스 규칙은 실패 테스트 먼저(이메일 중복 409·잘못된 자격 401 → red→green), 컨트롤러는 구현 후 통합 테스트로 검증했습니다.

**T18. UserDetailsService 미도입 — Claude Code (설계 판단)**
- 무상태 JWT + 커스텀 로그인 엔드포인트에서는 Spring 의 `AuthenticationManager`/`UserDetailsService` 를 쓰지 않고, `AuthService` 에서 `PasswordEncoder.matches` 로 직접 대조했습니다. **이유**: 매 요청 인증은 JWT 필터가 이미 처리하며, 직접 대조가 더 단순하고 `common ← user` 경계도 깔끔합니다.
- 로그인 실패 시 **이메일 미존재와 비밀번호 불일치를 동일한 401(`INVALID_CREDENTIALS`)** 로 통일했습니다. **이유**: 계정 존재 여부가 노출되면 사용자 열거(enumeration) 단서가 됩니다.
- 컨트롤러 흐름의 401 은 필터단(`AuthenticationEntryPoint`) 이 아니라 `UnauthorizedException`(`BusinessException` 하위)으로 던져 `GlobalExceptionHandler` 를 거칩니다. **이유**: 서비스에서 `AuthenticationException` 을 던지면 advice 폴백(500)으로 빠지므로, 상태 캐리어 예외로 통일했습니다.

**T19. `/api/auth/me` 인증 누락 버그 발견·수정 — Claude Code (버그 수정, 통합 테스트로 발견)**

### [Phase 4B] 응답 조회 + 대시보드 집계 — Claude Code (하이브리드 TDD)

수집된 응답을 제작자가 보는 절반입니다. 집계 정의(완료율·추이 구간·타입별 통계)는 사용자가 선택했고, 구현·검증을 진행했습니다.

**T39. 차트가 요구한 것이 스키마에 없어 컬럼을 추가 — Claude Code (설계 판단)**
- 일별 추이를 "발행일부터"로 그리기로 정했는데 **발행 시각을 알 방법이 없었습니다.** `updatedAt` 은 이후 제목 수정에도 갱신되고, `createdAt` 을 쓰면 아직 응답을 받을 수 없던 작성 기간까지 차트에 들어갑니다. `V3` 로 `forms.published_at`·`closed_at` 을 추가하고 상태 전이 시점에 기록합니다. **이유**: 감사 컬럼은 "언제 수정되었는가"를 답하지 그 자리에서 "언제 발행되었는가"를 답하지 못합니다. 둘은 다른 질문입니다.
- `closed_at` 을 함께 둔 이유는 구간의 끝을 고정하기 위함입니다. 없으면 오래전 종료한 폼의 추이가 오늘까지 0 으로 끝없이 늘어납니다.
- 기록 위치는 엔티티의 `changeStatus` 안 — 상태와 시각이 같은 지점에서 바뀌므로 어긋날 수 없습니다. 스키마·엔티티·`docs/04` 3자 일치도 함께 맞췄습니다.

**T40. 완료율을 정의대로, 그러나 질의 한 번으로 — Claude Code (설계 판단)**
- 완료율은 "응답 1건당 답한 질문 수 / 전체 질문 수의 평균"입니다. 응답마다 비율을 구해 평균 내는 대신 **답한 (응답, 질문) 쌍의 총수를 `총 응답 수 × 전체 질문 수` 로** 나눴습니다. 값은 같으면서 질의가 한 번이고, 답변 행이 하나도 없는 응답도 분모에 남아 0 으로 반영됩니다.
- 이때 저장 구조가 "값 하나가 한 행"이라 **체크박스가 같은 쌍에 여러 행을 만듭니다.** 중복을 제거하지 않으면 완료율이 1.0 을 넘습니다. 이 경우를 통합 테스트로 못박았습니다.

**T41. "없는 것"은 질의가 알려주지 않는다 — Claude Code**
- 응답이 없던 날과 아무도 고르지 않은 선택지는 **집계 결과에 행 자체가 없습니다.** 그대로 내려보내면 차트에서 그 날·그 선택지가 통째로 사라져, "0 이었다"가 아니라 "존재하지 않았다"로 보입니다. 폼 구조를 아는 서비스가 0 으로 메웁니다.
- 날짜 채우기를 DB 에 맡기지 않은 이유는 `generate_series` 가 PostgreSQL 전용이라 H2 테스트가 깨지기 때문입니다.

**T42. 이식성 규칙이 실제로 세 번 걸림 — Claude Code (버그 수정)**
- 통합 테스트에서 실제 SQL 로 세 가지가 드러났습니다. 모두 단위 테스트(모의 객체)만으로는 잡히지 않는 종류입니다.

  - 응답 삭제 시 이미 읽어 둔 `Answer` 들이 삭제된 `Response` 를 참조한 채 flush 되어 실패. → **DB 의 ON DELETE CASCADE 는 데이터 정합을 지킬 뿐 영속성 컨텍스트를 정리해 주지 않는다**는 점을 확인하고, 답변 행을 명시적으로 먼저 삭제. 폼 삭제 때 통했던 "DB 캐스케이드 위임"을 그대로 옮기려다 드러난 차이입니다.

**T43. 실측으로 집계 값 검증 — Claude Code**
- 로컬 PostgreSQL 에서 질문 4종·응답 3건(4문항·2문항·2문항 응답)을 만들어 확인했습니다: 완료율 `(4+2+2)/(3×4) = 0.667`, 평점 평균 `(5+3+4)/3 = 4.0`, 선택형 분포에 미선택 선택지 `불만족:0` 포함, 체크박스가 상세에서 한 문항(`["가","다"]`)으로 묶임, 무응답 문항 `answered=false`.

### Phase 6 — 리프레시 토큰 · 프론트 인프라 · 인증 화면

**T51. 프론트를 짜기 전에 백엔드의 빈칸부터 확인 — Claude Code (설계 판단)**
- 프론트의 401 처리 정책을 정하려면 토큰 갱신 수단이 있어야 하는데, `src/` 전체에 `refresh` 문자열이 **0건**이었습니다. 액세스 토큰 1시간이 전부였으므로 현재 설계로는 **1시간마다 강제 재로그인**이고, 로그아웃도 클라이언트가 토큰을 버리는 것이 전부(서버 무효화 불가)였습니다.
- 프론트 작업을 멈추고 백엔드에 리프레시 토큰을 먼저 넣었습니다. 무상태 JWT 는 폐기가 불가능하므로, **"갱신할 권리"만 DB 상태로** 남기는 방식입니다. 인증 자체는 그대로 무상태이고(요청마다 DB 를 보지 않습니다) 상태를 갖는 것은 갱신 경로뿐입니다.

**T52. 해시 알고리즘을 용도로 갈랐음 — Claude Code (설계 판단)**
- 토큰 원문 대신 해시를 저장하되 **비밀번호와 달리 BCrypt 를 쓰지 않았습니다.** 리프레시 토큰은 **해시로 조회**해야 하는데 BCrypt 는 행마다 salt 가 달라 동등 비교가 불가능해 전 행 스캔이 됩니다. 원문이 `SecureRandom` 256비트라 사전 공격 대상이 아니므로 key stretching 이 주는 이득도 없어 SHA-256 을 골랐습니다.
- 사용한 토큰을 **삭제하지 않고 `revoked_at` 으로 남기는** 것도 의도적입니다. 행을 지우면 "회전된 토큰"과 "존재한 적 없는 토큰"을 구분할 수 없어 재사용 탐지 자체가 불가능해집니다.

**T53. 탐지가 401 만 돌려주고 아무것도 무효화하지 못했음 — Claude Code (버그 수정)**
- 회전·재사용 탐지를 구현하고 테스트 148개가 모두 초록이었지만, 런타임으로 기동해 DB 를 열어 보니 **재사용 탐지 후에도 다른 토큰의 `revoked_at` 이 비어 있었고 실제로 갱신이 계속 성공**했습니다. 즉 탐지가 401 만 돌려주고 아무 일도 하지 않는 상태였습니다.
- 원인은 트랜잭션 경계였습니다. 전량 폐기(UPDATE) 직후 `UnauthorizedException` 을 던지는데, 런타임 예외의 기본 규칙대로 **롤백이 걸리면서 그 UPDATE 가 함께 취소**되었습니다.
- 중요한 것은 **두 테스트 계층이 모두 이 버그를 놓쳤다**는 점입니다. 단위 테스트는 레포지토리가 목이라 "호출되었는가"만 볼 수 있고 커밋 여부는 볼 수 없으며, 기존 통합 테스트는 `@Transactional` 로 요청들이 한 트랜잭션을 공유해 애초에 롤백이 일어나지 않습니다. 그래서 `@Transactional` 없이 실제 커밋이 일어나는 테스트를 따로 두고, 수정 전 red 를 확인한 뒤 고쳤습니다.
- 해결은 `noRollbackFor` 를 호출 계층마다 붙이는 대신 `revokeAllByUserId` 를 `REQUIRES_NEW` 로 분리했습니다. **이유**: 폐기의 보증이 호출자의 종료 방식에 의존하면 새 호출자가 생길 때마다 같은 함정을 반복합니다.

**T54. 프론트에서 같은 규칙이 반대편 함정을 만듦 — Claude Code (설계 판단)**
- 리프레시 토큰이 1회용이라는 서버 규칙이 클라이언트에 그대로 함정이 됩니다. 대시보드처럼 화면 진입과 동시에 여러 쿼리가 나가는 곳에서 토큰이 만료되면 **동시에 여러 요청이 401 을 받아 갱신을 각자 호출**하고, 두 번째부터는 "이미 회전된 토큰"이 되어 서버의 재사용 탐지를 오작동시킵니다. 정상 사용자의 세션이 통째로 끊깁니다.
- 진행 중인 갱신 Promise 를 공유하는 **single-flight** 로 막았습니다. 서버의 보안 장치가 클라이언트의 동시성과 만나 역효과를 내는 자리라, 양쪽을 함께 봐야 나오는 결론이었습니다.

**T55. 실패 문구를 프론트에서 다시 쓰지 않음 — Claude Code (설계 판단)**
- 서버가 보낸 `message` 를 그대로 화면에 씁니다. 프론트에서 문구를 다시 쓰면 같은 규칙이 두 곳에 존재하게 되는데, 특히 로그인 실패는 서버가 **이메일 없음과 비밀번호 틀림을 일부러 구분하지 않는**(계정 열거 차단) 결정이라 화면에서 친절하게 쪼개는 순간 그 의도가 깨집니다.
- 대신 400 의 `fieldErrors` 는 필드→사유 맵으로 펴서 해당 입력칸 옆에 붙이고, 필드로 특정되지 않는 실패(409 이메일 중복)만 배너로 보여 줍니다. `traceId` 도 화면에 작게 노출해, 사용자가 그 값만 알려 줘도 서버 로그에서 요청을 바로 찾을 수 있게 했습니다.

**T56. CORS 를 열지 않고 프록시로 우회 — Claude Code (설계 판단)**
- 프론트(5173)와 백엔드(8080)를 잇는 방법으로 백엔드에 CORS 설정을 추가하는 대신 **Vite dev 프록시**(`/api` → 8080)를 택했습니다. 브라우저 입장에서 동일 출처가 되어 백엔드 변경이 0이고, 인증 헤더·자격증명 정책을 건드릴 일이 없어 표면이 더 작습니다. 배포 시에는 리버스 프록시가 같은 역할을 합니다.

**T57. 실측으로만 드러난 표기 하나 — Claude Code**
- 문서용 예시를 캡처하다 **갱신 응답의 `accessToken` 이 직전 로그인과 문자열까지 동일**한 것을 발견했습니다. JWT 페이로드가 subject·iat·exp 뿐이고 iat/exp 해상도가 초라서 같은 사용자가 같은 초에 발급받으면 서명까지 같아집니다. 1초 뒤 재확인해 값이 달라지는 것까지 보고 `docs/05` 에 사실로 적었습니다 — 유효성 문제는 아니지만 "갱신했는데 토큰이 그대로"로 보일 수 있는 자리입니다.

**T58. 스캐폴딩 정리와 strict 를 지금 — Claude Code**
- `tsconfig.app.json` 에 **`strict` 가 빠져 있었습니다**(Vite 8 템플릿 기본값). API 응답을 다루는 코드가 들어오기 전에 켰습니다 — 나중에 켜면 그때까지 쌓인 null 처리 누락을 한꺼번에 고쳐야 합니다. Vite 템플릿 자산(로고·데모 CSS)도 함께 지웠습니다. 쓰지 않는 파일이 남아 있으면 어디까지가 우리 코드인지 흐려집니다.
- 루트 `.env.example` 의 `VITE_API_BASE_URL` 은 **Vite 가 읽지 않는 위치**였습니다(Vite 는 프로젝트 디렉터리 기준). 프론트 변수는 `open_forms_frontend/.env.example` 로 옮기고 루트에는 안내만 남겼습니다.

### Phase 7 — 폼 목록 · 폼 빌더

**T59. 계획서의 "5종"을 따르지 않고 9종 — Claude Code (설계 판단)**
- 작업 계획에는 질문 유형 5종으로 적혀 있었지만, 백엔드는 9종을 구현했고 `docs/05` 도 9종을 계약으로 명시하고 있었습니다. 화면에서 일부만 노출하면 **문서·코드·화면이 어긋납니다.** 계획서보다 이미 확정된 계약을 따랐습니다.

**T60. 빈 상태를 두 갈래로 — Claude Code (설계 판단)**
- "폼이 하나도 없음"과 "필터에 걸리는 폼이 없음"을 다른 화면으로 나눴습니다. 뭉뚱그리면 폼을 여러 개 만든 사용자가 상태 필터를 좁혔을 때 "아직 만든 폼이 없습니다"를 보고 **데이터가 사라졌다고 오해**합니다. 각각 다음 행동도 다릅니다 — 앞은 "첫 폼 만들기", 뒤는 "전체 보기"입니다.

**T61. 서버가 막을 일을 화면이 먼저 알림 — Claude Code (설계 판단)**
- 질문 편집은 `DRAFT` 에서만 가능합니다(409 `FORM_NOT_EDITABLE`). 버튼을 두고 눌렀을 때 409 를 보여 주는 대신, 편집 UI 자체를 없애고 **왜 못 고치는지**를 배너로 설명했습니다.
- 반대로 **제목·설명은 발행 후에도 수정 가능**합니다. 소스를 확인해 `FormService.update` 에는 `requireEditable()` 이 없다는 것을 보고, 실제로 발행된 폼에 `PUT` 을 보내 200 을 확인한 뒤 그대로 화면에 반영했습니다. 수집된 응답의 의미를 바꾸지 않으므로 막을 이유가 없는 비대칭이며, 추측으로 함께 잠갔다면 없는 제약을 만들 뻔했습니다.

**T62. 화면 검증의 위치를 명시 — Claude Code (설계 판단)**
- 선택지 2개 이상·min≤max 같은 규칙을 화면에도 뒀습니다. 서버 규칙과 중복이지만, 없으면 사용자가 저장을 눌러야만 400 으로 배웁니다. 다만 이것은 **안내용이고 최종 판정은 서버**라는 경계를 코드 주석과 함수 이름(`draftProblem`)에 남겨, 나중에 서버 규칙이 바뀌었을 때 화면 검증이 진실인 척하지 않도록 했습니다.
- 실측으로 대조했습니다: 선택지 1개 → 400 `OPTIONS_REQUIRED`, 발행 후 질문 추가 → 409 `FORM_NOT_EDITABLE`.

**T63. 편집 상태를 서버 표현과 다르게 설계 — Claude Code**
- 범위(min/max)를 **문자열로** 들고 있습니다. 숫자 타입에는 "비우는 중"이라는 중간 상태가 없어서, 숫자로 두면 입력칸을 지우는 순간 0 이 되거나 NaN 이 됩니다. 선택지에는 **클라이언트 전용 key** 를 붙였습니다 — 새 선택지에는 서버 id 가 없고 label 은 중복될 수 있어 렌더링 key 로 쓸 수 없습니다.
- 유형을 바꿔도 입력값은 남기되 **저장 시점에 해당 없는 값을 걸러** 보냅니다. 서버는 무시하도록 되어 있지만(`docs/05`), 남은 값이 저장되면 나중에 유형을 되돌렸을 때 옛 값이 되살아납니다.

**T64. 되돌릴 수 없는 동작에 이유를 붙임 — Claude Code (설계 판단)**
- 발행·마감은 단방향 전이라 실수하면 복구 수단이 없습니다. `window.confirm` 대신 `<dialog>` 기반 확인창을 두고 **무엇이 되돌릴 수 없는지**를 문장으로 적었습니다(발행 후에는 질문 수정 불가, 마감 후에는 응답 중단). 기본 confirm 은 서식을 담을 수 없고 브라우저마다 모양이 다릅니다.
- 질문이 0개인 폼의 발행은 화면에서 막았습니다. 서버 규칙은 아니지만 응답자가 빈 화면을 보게 되는 상황이 의도일 리 없습니다.
