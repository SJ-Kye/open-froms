# 07. 미완성 / 개선하고 싶은 점

## 현재 설계의 한계

| 영역 | 내용 |
|---|---|
| **예외 처리** | `GlobalExceptionHandler`가 `ResponseEntityExceptionHandler`를 상속하지 않아 일부 클라이언트 오류(경로 변수 타입 불일치, Enum 변환 실패, 405)가 500으로 응답. `ResponseEntityExceptionHandler` 상속으로 해결 가능 |
| **JWT 시크릿** | `application.yml`에 fallback 기본값 존재. 환경변수 미설정 시 조용히 기동 → 기동 시점 실패로 교체 필요 |
| **요청 제한 없음** | `/login`·`/register`·공개 응답 제출에 rate limiting 없음. 무차별 대입·응답 오염 가능 |
| **토큰 저장소** | refresh_token을 `localStorage`에 저장 → XSS 성공 시 노출. `httpOnly` 쿠키로 전환 여지 |
| **정렬 파라미터** | `Pageable` 직접 바인딩 → 임의 프로퍼티 경로 허용. 화이트리스트 필요 |
| **DB 불변식** | `answers`의 "값 컬럼 정확히 하나" 규칙이 코드에만 존재. CHECK 제약 부재. `position` 유니크 제약도 없음 |
| **시각 타입 혼재** | 리소스 타임스탬프는 `LocalDateTime`(오프셋 없음), 에러는 `Instant`. 일별 집계가 서버 로컬 날짜 기준 → `timestamptz` + `Instant` 통일 필요 |
| **로그 증가** | `api_call_logs`·`refresh_tokens` 만료 행 정리 정책 없음 |
| **대시보드** | DATE/TIME 질문 구간 집계·기간 필터·CSV 내보내기 미구현. 차트에 접근성 대체 표현 없음 |

## 알려진 버그

**문항 순서 변경이 저장되지 않습니다.** `savePlan.ts`의 변경 판정이 `saved.position` 대신 화면 인덱스를 비교해 순서 변경이 항상 0건으로 집계됨 → 저장 바 미표시.

**다른 탭에서 추가한 문항이 저장 시 삭제될 수 있습니다.** 편집 중 창 포커스 재조회가 동작해 로컬 상태에 없는 문항이 삭제 대상으로 분류됨.

## 테스트 불균형

백엔드 148개 테스트(도메인·통합·문서 정합)에 비해 **프론트엔드 테스트 0건**. `savePlan.ts`·`questionDraft.ts`·`answerDraft.ts`는 순수 함수라 테스트 비용이 낮음에도 비어 있고, 위 순서 변경 버그가 이 공백에서 살아남음.

## 시간이 더 있었다면

**순서 변경 버그 수정 + 프론트 단위 테스트 도입** — 가장 먼저 하고 싶은 작업입니다.

*못한 이유*: 백엔드 도메인 규칙(상태 전이, 응답 검증 8종, 리프레시 토큰 회전)의 테스트를 먼저 채우는 데 시간을 썼고, 프론트는 화면을 손으로 눌러 확인하는 방식으로 대체했습니다. 순서 변경은 "드래그 → 저장 → 새로고침"까지 해야 드러나는데 수동 확인에서는 드래그 직후 화면만 보고 넘어가 끝까지 발견하지 못했습니다.

*어떻게 할 것인가*: (1) `savePlan.ts`의 `diffQuestions`를 `saved.position`과 `draft.position`을 비교하도록 고치고, 화면 인덱스는 렌더링에만 쓰도록 경로를 분리합니다. (2) Vitest를 도입해 `savePlan.ts`·`questionDraft.ts`·`answerDraft.ts` 세 순수 함수에 먼저 붙입니다 — 순서만 바꿨을 때 변경 1건, 내용만 바꿨을 때 1건, 둘 다 바꿨을 때 1건, 무변경 0건을 회귀 테스트로 고정합니다. (3) 다른 탭 충돌은 포커스 재조회 결과를 로컬 draft와 머지하지 않고 "서버에 새 문항이 생겼습니다" 안내 후 사용자가 재조회를 선택하게 바꿉니다. (4) 그 위에 Testing Library로 FormBuilder 저장 플로우 통합 테스트 1개를 얹어 두 버그가 다시 살아나지 못하게 막습니다.
