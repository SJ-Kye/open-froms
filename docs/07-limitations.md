# 07. 미완성 / 개선하고 싶은 점

이 문서는 **알면서 남겨 둔 것**과 **만들고 나서 발견한 것**을 구분해 적습니다. 다른 문서에서 "개선 여지" 로 넘긴 항목들이 모두 여기로 모입니다.

## 현재 설계의 한계

### 1. 예외 처리 — 클라이언트 오류 일부가 500 으로 나갑니다

`GlobalExceptionHandler` 가 `ResponseEntityExceptionHandler` 를 상속하지 않은 채 `@ExceptionHandler(Exception.class)` 를 두고 있습니다. `ExceptionHandlerExceptionResolver` 가 스프링 MVC 의 기본 예외 해석(`DefaultHandlerExceptionResolver`) 보다 먼저 실행되므로, 프레임워크가 400·405 로 변환했어야 할 예외를 이 catch-all 이 먼저 가로채 `500 INTERNAL_ERROR` 로 응답합니다.

| 요청 | 현재 | 기대 |
|---|---|---|
| `GET /api/forms/abc` (경로 변수 타입 불일치) | 500 | 400 |
| `GET /api/forms?status=BOGUS` (Enum 변환 실패) | 500 | 400 |
| `GET /api/auth/login` (미지원 메서드) | 500 | 405 |

**왜 놓쳤는지**를 적어 둡니다. 같은 성격의 `NoResourceFoundException`(매핑된 핸들러 없음)은 명시적으로 처리해 404 로 내리고 있습니다. 즉 문제의 종류는 인지했으나 그 한 사례만 막았습니다. 그리고 통합 테스트가 **정상 경로의 잘못된 값**(`@Valid` 실패, 도메인 규칙 위반)은 촘촘히 검증하는 반면 **경로·파라미터 자체가 깨진 요청**은 한 건도 없어, 스위트의 사각지대와 결함이 정확히 겹쳤습니다.

개선 방향은 `ResponseEntityExceptionHandler` 상속으로 프레임워크 예외의 기본 처리를 받되 본문만 공통 포맷으로 덮어쓰는 것입니다. 이때 400·405·415 각각에 에러 코드를 부여해야 하므로 [05. API 설계](05-api-design.md) 의 에러 코드 일람과 동반 갱신이 필요합니다.

### 2. 인증 — 운영 전제로 넘어가기에는 모자란 지점들

- **JWT 시크릿에 기본값이 있습니다.** `application.yml` 이 `${JWT_SECRET:change-me-...}` 로 fallback 하므로, 환경변수를 잊은 채 기동해도 애플리케이션이 정상적으로 뜹니다. 이 저장소를 읽을 수 있는 사람은 누구나 토큰을 위조할 수 있습니다. 로컬 실행 편의를 위한 선택이었지만 **실패해야 할 상황에서 조용히 성공하는 쪽**이라 방향이 틀렸습니다. 기본값을 없애고 `@Validated` + 길이 검증으로 기동 시점에 실패시키는 편이 맞습니다.
- **회원가입이 계정 존재 여부를 알려 줍니다.** 로그인은 이메일 오류와 비밀번호 오류를 `INVALID_CREDENTIALS` 하나로 통일해 계정 열거를 막았는데, 같은 컨트롤러의 `POST /api/auth/register` 는 `409 EMAIL_ALREADY_EXISTS` 를 그대로 돌려줍니다. 위협 모델이 엔드포인트마다 다른 셈입니다. 다만 가입 화면에서 "이미 가입된 이메일" 안내를 없애면 사용성이 크게 나빠지므로, 실제 해법은 응답을 뭉개는 것이 아니라 아래의 요청 제한과 함께 다루는 것입니다.
- **요청 제한(rate limiting)이 없습니다.** `/login`·`/register`·`/refresh` 에 시도 횟수 제한이 없어 온라인 무차별 대입을 늦출 수단이 없습니다. 익명 제출(`POST /api/public/forms/{slug}/responses`) 도 중복 방지·캡차·IP 제한이 모두 없어, 공개 링크를 아는 사람이 응답을 무제한으로 채워 넣을 수 있고 그 결과가 대시보드 집계를 그대로 오염시킵니다.
- **토큰을 `localStorage` 에 둡니다.** XSS 가 성립하면 14일짜리 리프레시 토큰까지 읽힙니다. 회전·재사용 탐지가 피해 범위를 줄여 주긴 하지만 예방책은 아닙니다. 리프레시 토큰만 `httpOnly` 쿠키로 옮기는 절충안이 있었는데, 프론트 배포 형태(동일 출처)가 확정되기 전에 프론트 인증 인프라를 먼저 만든 순서 때문에 선택지가 좁아졌습니다.
- **정렬 파라미터가 열려 있습니다.** 목록 API 가 `Pageable` 을 그대로 바인딩하므로 `?sort=user.passwordHash,asc` 같은 프로퍼티 경로가 통합니다. 해시 값을 반환하지는 않지만 정렬 결과가 값을 추론하는 단서가 됩니다. 허용 프로퍼티 화이트리스트가 필요합니다.

### 3. 데이터베이스 — 불변식이 코드에만 있습니다

- **`answers` 의 "값 컬럼 중 정확히 하나" 규칙에 DB 제약이 없습니다.** 스키마 주석과 [04. DB 설계](04-db-design.md) 는 이 불변식을 단언하지만, 실제로 지키는 것은 `Answer` 의 생성자뿐입니다. 애플리케이션을 거치지 않은 수정이나 향후의 배치 작업이 규칙을 깰 수 있습니다. PostgreSQL·H2 양쪽에서 동작하는 `CHECK` 로 표현 가능한 규칙이라 미룰 이유가 없었습니다.
- **`questions(form_id, position)` 과 `question_options(question_id, position)` 에 유니크 제약이 없습니다.** 같은 position 을 가진 질문이 둘 생기면 `ORDER BY position` 의 결과가 비결정적이 되고, 응답 상세 조립이 그 순서에 의존합니다. 제약을 걸지 않은 것은 순서 변경 시 전량 재부여 과정에서 일시적으로 값이 겹칠 수 있어서인데, 그렇다면 재부여를 음수 오프셋 경유 2단계로 바꾸고 제약을 거는 편이 맞습니다.
- **인덱스 하나가 중복입니다.** `forms(user_id)` 와 `forms(user_id, status)` 가 둘 다 있는데, 04 문서가 `ix_answers_question_option` 을 설명하며 스스로 제시한 최좌측 프리픽스 논리를 그대로 적용하면 복합 인덱스가 단독 조회까지 커버합니다. 같은 문서 안에서 한 인덱스에는 적용하고 다른 인덱스에는 적용하지 않은 판단 불일치입니다. `ix_forms_user_id` 는 제거 대상입니다.
- **시각 타입이 두 갈래로 섞여 있습니다.** 에러의 `timestamp` 는 `Instant`(UTC, `Z` 접미사)인데 리소스의 `createdAt`·`submittedAt` 은 `LocalDateTime`(오프셋 없음)입니다. DB 컬럼이 타임존 없는 `TIMESTAMP` 이기 때문인데, 더 실질적인 문제는 **일별 응답 추이가 서버 로컬 날짜로 버킷팅된다**는 점입니다. 서버 타임존이 바뀌거나 다른 지역에 배포되면 같은 데이터의 차트 모양이 달라지는데, 그 계약이 어디에도 적혀 있지 않습니다. `timestamptz` + `Instant` 로 통일하고 집계 기준 타임존을 API 계약에 명시하는 것이 옳습니다.
- **`api_call_logs` 가 무한히 증가합니다.** 모든 요청이 한 행을 남기지만 이를 읽는 엔드포인트도, 보존 기간도, 파티셔닝도 없습니다. `refresh_tokens` 도 로그인·갱신마다 한 행씩 쌓이고 만료·폐기된 행을 정리하는 작업이 없습니다(폐기 흔적은 재사용 탐지에 필요하므로 즉시 삭제가 아니라 **만료 후 유예 기간을 둔 정리**여야 합니다).
- **낙관적 락(`@Version`)이 없습니다.** 두 창에서 같은 폼을 수정하면 마지막 쓰기가 조용히 이깁니다. 단일 사용자 소유 리소스라 충돌 확률은 낮지만, 아래 4번의 실제 결함이 정확히 이 결여에서 파생됩니다.

### 4. 알려진 결함 (수정 예정)

설계상의 선택이 아니라 **버그**입니다. 구분해 적습니다.

- **문항 순서 변경이 저장되지 않습니다.** 빌더의 변경 판정(`savePlan.ts`)이 "서버로 보낼 형태끼리 비교" 라는 원칙 자체는 맞으나, 비교 기준을 만들 때 저장된 `position` 이 아니라 **화면의 새 인덱스**를 다시 넣습니다. 양변의 `position` 이 항상 같아지므로 순서 변경은 어떤 경우에도 변경으로 집계되지 않고, 「변경 N건」이 0 이라 저장 바 자체가 나타나지 않습니다. 사용자는 바뀐 순서를 보면서 저장할 방법이 없는 채로 화면을 떠나게 됩니다. 기준을 `saved.position` 으로 바꾸면 해결됩니다.
  - 이 결함은 **주석이 코드보다 앞서간 사례**이기도 합니다. `FormBuilderPage` 상단 주석이 "저장 시 화면 순서대로 position 을 전량 재부여하므로 순서 변경이 반영된다" 고 서술하는데, 코드가 그렇게 동작하지 않습니다. 순수 함수인 `buildSavePlan` 에 유닛 테스트가 하나라도 있었다면 즉시 드러났을 문제입니다(아래 5번).
- **다른 탭에서 추가한 문항이 저장 시 삭제될 수 있습니다.** 빌더는 편집 중일 때 서버 상태 재동기화를 건너뛰지만 React Query 의 창 포커스 재조회는 그대로 동작하므로, 백그라운드에서 갱신된 `form.questions` 에는 있고 로컬 편집 상태에는 없는 문항이 "삭제 대상" 으로 분류됩니다. 삭제 건수를 보여 주는 확인 창이 유일한 안전장치입니다. 편집 중에는 재조회 자체를 멈추거나, 저장 직전에 서버 버전을 확인하는 처리가 필요합니다.

### 5. 테스트 — 백엔드와 프론트엔드의 비대칭

백엔드는 148개 테스트가 도메인 규칙을 실패 테스트부터 고정하고, 문서·OpenAPI·소스의 3자 정합까지 `ApiContractConsistencyTest` 로 기계화했습니다. 반면 **프론트엔드에는 테스트가 한 건도 없습니다.** 러너조차 설치되어 있지 않습니다.

이것이 단순한 분량 문제가 아닌 이유는, 프론트의 비자명한 규칙이 이미 순수 모듈로 분리되어 있기 때문입니다 — `savePlan.ts`(변경 판정), `questionDraft.ts`(편집 상태 ↔ 요청 변환), `answerDraft.ts`(응답 작성 상태). 의존성 없이 호출 가능한 함수들이라 테스트 비용이 거의 들지 않는데도 비어 있고, 4번의 순서 변경 결함이 정확히 그 공백에서 살아남았습니다. 하이브리드 TDD 를 표방한 프로젝트에서 계층 하나만 규율 밖에 있었던 셈입니다.

그 밖의 공백입니다.

- **동시성 시나리오가 없습니다.** 리프레시 토큰 동시 회전, 마감과 제출이 겹치는 경우를 검증하지 않았습니다. 재사용 탐지의 롤백 문제는 결국 런타임에서 발견해 비트랜잭션 테스트를 신설해 고정했는데, 이는 트랜잭션 경계가 얽힌 규칙을 통합 테스트가 구조적으로 놓칠 수 있음을 보여 줍니다.
- **N+1 부재를 검증하는 테스트가 없습니다.** 배치 조회로 N+1 을 피하는 것이 반복된 설계 목표인데, 이를 지키는지 확인하는 것은 사람의 눈뿐입니다. 실제로 응답 상세 조립에서 선택지 라벨을 지연 로딩으로 건건이 읽는 N+1 이 남아 있습니다 — 하필 N+1 을 해결했다고 주석에 적어 둔 클래스입니다. 쿼리 카운터를 붙인 테스트가 있었다면 잡혔습니다.
- **정합 테스트 자체가 깨지기 쉽습니다.** 엔드포인트 개수를 `hasSize(20)` 으로 못 박아 두어 문서와 구현을 함께 늘리는 정당한 변경에도 실패하고, 에러 코드 수집이 대문자 문자열 리터럴 정규식 스캔이라 무관한 상수가 오탐을 냅니다. 양방향 비교가 이미 본질을 검증하므로 개수 단언은 없는 편이 낫습니다.

### 6. 대시보드 — 집계의 깊이

DATE·TIME 질문은 분포 없이 응답 수만 보여 줍니다. 날짜 응답을 의미 있게 집계하려면 구간(일·주·월) 결정이 필요한데 그 UI 를 만들지 못했습니다. 그 밖에 기간 필터, 질문 간 교차 분석, CSV 내보내기가 없어 "집계 결과를 보는" 것까지는 되지만 "집계 결과를 가지고 나가는" 것은 되지 않습니다.

접근성 측면에서도 차트가 SVG 뿐이라 스크린 리더 사용자가 읽을 대체 표현이 없고, 완료율 미터에 `role`·`aria-valuenow` 가 없습니다. 폼 오류를 입력과 연결하는 `aria-invalid`·`aria-describedby` 도 빠져 있습니다. `jsx-a11y` 린트 플러그인을 켜지 않아 이런 누락이 기계적으로 잡히지 않았습니다.

### 7. 그 밖에

- **순차 PK 가 노출됩니다.** 폼·질문·응답 id 가 `BIGINT IDENTITY` 라 URL 에 그대로 드러납니다. 소유권 검사가 모든 제작자 경로에 걸려 있어 열람 자체는 막히지만, id 를 훑으면 서비스 전체의 폼 생성량을 추정할 수 있습니다(정보 노출). `users` 에만 UUIDv7 을 쓴 것은 계정 식별자가 외부에 오래 남기 때문인데, 같은 논리를 공개 링크에 쓰이는 리소스로 넓힐 여지가 있습니다. 다만 공개 응답 경로는 이미 추측 불가능한 `slug` 를 쓰고 있어 우선순위는 낮습니다.
- **프론트에 `ErrorBoundary` 가 없습니다.** 질문 유형 변환기가 미지의 타입에 예외를 던지므로, 백엔드가 유형을 하나 추가하면 화면 일부가 아니라 앱 전체가 빈 화면이 됩니다.
- **`CLAUDE.md` 가 낡았습니다.** 삭제하기로 한 `answer_options` 조인 테이블을 포함한 7테이블 모델을 여전히 기술합니다. 커밋되는 `docs/` 는 정합을 유지했지만 개발 지침 파일은 갱신 대상에서 빠졌습니다.

## 시간이 더 있었다면 추가했을 것

우선순위 순으로 적습니다. 위쪽 세 가지는 "추가" 라기보다 **마감해야 할 것**입니다.

1. **순서 변경 결함 수정과 프론트 테스트 도입** — Vitest 를 붙이고 `savePlan`·`questionDraft`·`answerDraft` 부터 덮습니다. 결함 재현 테스트를 먼저 작성(red)하는 순서로 백엔드와 같은 규율을 맞춥니다.
2. **예외 처리 보정과 JWT 시크릿 기본값 제거** — 둘 다 한두 줄 규모이며, 각각 잘못된 상태 코드와 실패해야 할 기동을 바로잡습니다.
3. **DB 불변식의 제약 이관** — `answers` 의 CHECK, `position` 유니크, 중복 인덱스 제거를 `V5` 로 묶습니다.
4. **요청 제한과 중복 응답 방지** — 인증 엔드포인트에 시도 제한을, 공개 제출에 제출자 지문(쿠키·IP 해시) 기반 중복 억제를 둡니다. 익명성을 지키면서 집계 신뢰도를 올리는 선이 어디인지 결정하는 것이 실제 작업량입니다.
5. **시각 표기 통일** — `timestamptz` + `Instant` 로 맞추고 집계 기준 타임존을 API 계약에 명시합니다. 마이그레이션과 프론트 포맷터를 함께 손봐야 합니다.
6. **이력·토큰 테이블 운영 정책** — `api_call_logs` 보존 기간과 조회 API, `refresh_tokens` 만료분 정리 스케줄러.
7. **대시보드 확장** — 기간 필터, CSV 내보내기, DATE 질문 구간 집계, 차트의 표 대체 표현과 `jsx-a11y` 도입.
8. **기능 확장** — 파일 업로드 문항, 폼 템플릿, 응답 마감 조건(인원·기한 자동 마감), 이메일 알림. 비범위로 선언했던 항목들이며 도메인 모델은 이미 이들을 받아들일 여지가 있습니다.
